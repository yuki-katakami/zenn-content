## はじめに

あるIT企業でシステム運用部の課長をしています。前回の記事では、Claude Code で 3ヶ月で 17個のツールを作った話の裏で、結局いちばん効いたのは「**自分の時間を計測すること**」だったという話を書きました。

今回は、その 17個のツールに関する**前回の記事では触れなかった、もうひとつの側面**の話です。

17個のツール、振り返ってみると、その多くを「**自分だけが使える形**」ではなく「**いつか組織で使える形**」を意識して設計していました。

「組織で使える形」というのは、私の中ではこういうイメージです——**承認や判断のような責任を伴う部分は、それを負うべき人が手で行う。その手前までの作業だけを自動化する**。そうして、利用者を段階的に増やしていく、というやり方です。

正直に書いておくと、**ここまでの方向性を言葉にできるようになったのは、Claude Code を使い始めてから**です。SikuliX や OCR や Power Automate を試していた頃は、「業務をもっと楽にしたい」という曖昧な思いはあっても、目指す絵を言語化することすらできていませんでした。**下から小さく改善し、その結果を報告し、今のシステムの不便な部分を共有しながら、より良い形へ変えていく一石になれたら**——そう思えるところまで来たのは、ごく最近のことです。

ただし、**現時点で組織展開できているわけではありません**。Claude Code の業務利用そのものがまだ**個人ベースで進めている段階**であること、まだ段階的に作っている最中であることもあって、ほとんどのツールは私自身が使っているのが実態です。一部だけを他の方に渡しているにとどまっています。**組織展開は、これからの話です**。

それでも、設計段階から「いつか他の人に渡せる形」にこだわっていたのは事実です。なぜそんなことにこだわったのか——前回は時間の話で記事を埋めてしまったので、その問いには触れていませんでした。

理由のひとつは、運用エンジニアとして長年働いてきた中で「**属人化を避ける**」という動作が当たり前として身についている、という側面です。それも本当です。

ただ、もうひとつ、もっと根本的な動機があります。

それは——**私は本当はチームで解決したかった。でも、何度試みてもチームを作れなかった**——という、ちょっと情けない経緯です。

## チームで解決したかった、でも作れなかった

私の所属しているのは、**システム運用保守を主事業とする中小のIT企業**です。社員の大半は顧客先に常駐しています。

会社にとって、チームを組んで稼働させていい活動は、基本的には**売上に直結する業務**です。これは中小のSES型企業としては自然な経営判断で、一概に否定できるものではありません。ただ、その前提に立つと、**バックオフィスや内製改善のためのチーム稼働は、どうしても優先順位が下がります**。

管理職として現場を見ていると、内製改善でやるべきことはたくさんあります。手作業で繰り返している業務、属人化したナレッジ、もう少しデータを整えれば判断が早くなる領域。同じことを感じている管理職は、私以外にも何人もいると思います。

それでも、その課題に対して「**チームを組んで本気で取り組む**」体制を作ることは、なかなかできませんでした。売上に直結しない領域に、稼働の取れる人材を割り当てる判断をするのは、組織として簡単ではありません。

正直に書いておくと、ここには**私自身の力不足の側面**もあります。本来、管理職である私はもっと**組織変化を促す活動**——文化の方を変えていく動きに労力をかけるべきだったのかもしれません。ただ、組織文化を変えるというのは、思っている以上に大きな労力が必要で、本業を回しながらそれをやり抜くのは、私には簡単ではありませんでした。

つまり、「**人がいないから**」だけでは済まない、**もっと根本的な、組織文化の側にある課題**がそこにあります。それを認識しながらも、私は完全には手をつけきれていない——というのが、いま振り返って言える正直なところです。

その代わりに私が選んだのは、「**自分一人でできる範囲で、何かを変え続ける**」という道でした。

## だから一人で試し続けてきた——SikuliX、OCR、Power Automate

そういうわけで、私は何度も「**自分一人でできる方法**」を試してきました。チーム化が無理ならせめて自分の手の中で、内製改善の試行錯誤を続けてきた、というのが実態です。

最初に試したのは、**SikuliX という RPA ツール**を使った自動化でした。SikuliX は画像認識ベースで画面要素を見つけて操作するツールで、Python (Jython) でスクリプトを書けます。

社内の業務システムにアクセスしてデータを取ってくる、画面操作を自動化する、というアプローチです。SikuliX を選んだのは、**自分にとって比較的書きやすかった**ことと、**自分一人でも面倒を見られるかもしれない**と思ったからです。実際、動くものはできました。一部の業務は、目に見えて楽になりました。

ただ、**UI が変わるたびに止まる**のです。SikuliX は画像認識で要素を見つけているので、ボタンの色や位置が少し変わるだけで反応しなくなります。業務システムのアップデート、画面構成の変更——そのたびに修正対応が必要で、私一人では追いきれませんでした。動かない時間と、保守する時間。両方を背負い続けるだけの余裕は、本業を回している管理職にはありません。やがて諦めることになりました。

次に試したのは、**OCR を組み合わせたブラウザ自動操作**でした。稟議フローの一部を自動化する目的で作ったものです。

これは、最初の RPA より少し長く生き延びました。ある程度の期間、ちゃんと使えていました。でも結局、業務システム側の構造変化のタイミングで、また使えなくなりました。理由は最初の挑戦とほとんど同じでした。**変化に弱い自動化は、メンテナンスの重みで自分を潰す**ということを、二度学ばされたわけです。

そして、もうひとつ大事な気づきがありました。**結果として使わなくなった**最大の理由は、メンテナンスの重さだけではありません。**動かなくなったツールを直すより、手動のほうがその場をしのぎやすい**のです。「動かないツールを直す時間」と「とりあえず手で対応する時間」を比べると、後者のほうが短く、確実で、判断も少なく済みます。

実感として、**自動化を維持するコストが手作業を上回った瞬間に、自動化は死ぬ**——ということを、私はこの2回で身をもって理解しました。

その後、社内が **Microsoft 365 に徐々に移行していく**流れが来ました。Power Automate を触るようになったのもこのタイミングです。そしてこの流れの中で、**AI を業務で使うことの検討と、その重要度が、じわじわと自分の中で増していきました**。

いま振り返ると、ずっと共通していたのは「**チームを作れない、でも何かは変えたい**」という、少し不器用な動きでした。何度試しても、メンテナンス工数の壁にぶつかって挫折する。それでも、また次のやり方を試す——という繰り返しでした。

## Claude Code との出会いが、何を変えたか

2026年から、Claude Code を業務の中心ツールとして本格的に使い始めました。3ヶ月で17個のツールが運用に乗った、というのは前回の記事で書いたとおりです。

ただ、**「速くツールが作れる」というのは、私にとっての本質ではありませんでした**。

これまでの試みを振り返ると、ツールが作れること自体は問題ではなかったのです。SikuliX も OCR + ブラウザ自動操作も Power Automate も、それぞれの時点で「動くもの」は作れていました。問題は、**作った後の維持・メンテナンス・引き継ぎ可能性**のところで、いつも力尽きていたという点です。

Claude Code を使い始めて、私の側で何か変わったとすれば、それは**ツールを作る「前提条件」のところ**でした。

ひとつ目は、**メンテナンスを一人で背負わずに済むようになった**ことです。仕様変更が起きたとき、UI が変わったとき、エラーが出たとき、Claude Code に状況を共有しながら一緒に修正案を考えられる。これは「動かなくなったらそこで死ぬ」という、これまでのツール群とは決定的に違う構造でした。

ふたつ目は、**作る段階から「他の人に渡せる形」を意識できるようになった**ことです。

正直に書くと、これまでの私は「自分一人で動けば OK」で作ってきました。エラーメッセージを丁寧に書く、コマンド名を覚えやすく整える、簡単なドキュメントを添える、といった作業は「あとでやろう」のままでした。一人で使う限りは、それで済みます。

ただ、これから他の人に渡していく段階に入るなら、**最低限のエラー対処ができる状態**にしておくことは必要になります。Claude Code は、「他の人に渡したい」と意識して頼めば、こうした作業を「ついでに」整えてくれます。一人で書いているときには後回しになるこれらを、**設計の段階から取り込める余裕**ができたこと——それが2つ目の変化です。

みっつ目は、**「これ、組織に展開できる形にしておこう」という選択肢を、初めて現実的に持てるようになった**ことです。これまでは自分一人でなんとか動かすので精一杯で、「組織に渡す」ところまで頭が回らない状態でした。Claude Code で作るときは、設計の時点から「いつか他の人に渡せる形」を意識する余裕ができました。

ここまで書くと Claude Code を持ち上げているように読めるかもしれません。少し正直に書いておくと、**Claude Code は万能ではないし、すべてが楽になったわけでもありません**。指示が曖昧だと的外れな出力が返ってきますし、生成されたコードを鵜呑みにすると後で痛い目を見ます。**Claude Code が機能するのは、こちら側に「何を作りたいか」「何を作りたくないか」がはっきりしている時だけ**です。

ただ、これまでの試みでは「動くものが作れても、それが死なないように維持する」ところで詰んでいた私にとって、その維持の部分が一段軽くなったことの意味は、思っている以上に大きかった——という、それだけの話でもあります。

もちろんこれは、**管理職で、チームを作れず、一人でツールを書き続けてきた私の状況だからこそ強く見えた変化**かもしれません。同じ Claude Code を使っても、立場や環境で感じる変化は違うのだろうと思います。

3ヶ月で17個のツール、というのは、結果としての数字に過ぎません。本当に変わったのは、**「動くものを作ること」から「組織で渡せるものを作ること」へ、自分の発想がアップデートされた**——その内側の変化のほうだったと、いま振り返って思います。

## 同じような立場の管理職へ

ここまで読んでくださった方の中に、**自分の組織でも内製改善のチームが作れない**、と感じている管理職の方がいるかもしれません。

私の話は、その状況に対する「解決策」ではありません。**一つの選択肢**として、こんなやり方もあったよ、という記録に近いものです。

ひとつだけ書いておきたいことがあります。「**チームが作れない**」のは、**誰の責任とも言いがたい構造の問題**だ、ということです。組織の事業構造、優先順位の決まり方、稼働の埋め方。多くの場合、それらは個人の管理職の力ですぐに変えられるものではありません。本来なら組織文化の方を変えていくべきだ、というのは正論ですが、それには大きな労力と時間が必要で、本業を回しながら片手間でやれるものではないのが現実です。

そんな状況で、それでも何かを変えたいと思っているなら、**AI を使った個人完結の試行錯誤は、いままでより一段現実的な選択肢になっています**。少なくとも私の経験では、SikuliX や OCR や Power Automate の頃とは、明らかに違う手応えがあります。

もちろん、これで組織が変わるわけではありません。私自身、まだ組織展開はこれからです。でも、「**動くものを作って、いつか他の人にも渡せる形にしておく**」という選択肢を持てることは、何もできない感じよりはずっとましな状況だと思っています。

そして、もう一つ。**自分一人で何もかもを抱え込む必要はありません**。Claude Code は道具です。ただ、これまでのスクリプト群と違って、**メンテナンスの時間に呼び出せる相手がいる**ということが、一人で詰まる時間を確実に減らしてくれます。これは、何度も力尽きて諦めてきた私のような人間にとって、思っていた以上に大きな違いでした。

## おわりに

前回の記事で書いた「**自分の時間を計測する**」という話と、今回書いた「**一人で組織に渡せるツールを作る**」という話。一見すると別々のテーマですが、私の中では同じ場所から出ています。

どちらも、「**与えられた範囲だけをこなすのではなく、自分にできる範囲を地道に広げていきたい**」という、たぶん 8年くらいずっと変わっていない動機から来ています。8年前にこの会社に入った時の私は、こんなツールを作ることになるとも、こんな記事を書くことになるとも、想像していませんでした。それでも、その時々の自分の役割と立場で「何を届けられるか」を考え続けてきた延長線上に、いまの景色があります。

これからどうしていきたいかは、自分の中でもまだ言葉になりきっていません。確かなのは、AI を道具にして自分にできる範囲を広げる試みを、これからも続けていくだろう、ということです。組織展開できる形に、もう少し近づけていきたい。組織文化の方を変えていくのは、**当面は私の手の届く範囲を超えています**。それでも、ツールを通じて現場で見えている課題を共有していくことが、**いつか組織の側の動きにつながるきっかけ**になるかもしれない——そのくらいの距離感です。

そして、同じような壁にぶつかっている管理職の方が、もし少しでもこの話に共感してくれたなら、それだけで、**この話を共有してよかったと思います**。
