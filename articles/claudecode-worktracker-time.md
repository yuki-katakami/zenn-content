## はじめに

あるIT企業でシステム運用部の課長をしています。39歳。プログラミングは独学で、業務で書くスクリプトはあれど「ソフトウェアエンジニア」を名乗ったことはありません。

2026年から、Claude Code を業務に本格導入しました。3ヶ月で17個の業務自動化ツールを作り、運用に乗せました。結果として工数は減り、空き時間ができ、新しいことに手を出す余裕も生まれました。

——という話を期待して開いてくれた方には申し訳ないのですが、この記事はそれが本題ではありません。

本題は、**「ツールを作り終えた後に、自分の時間が本当に意味あることに使えているのか分からなくなった」** という違和感と、その違和感を解消するために作った別の小さなツールの話です。

## 自動化したものたち

最初に背景を共有します。Claude Codeで作ったのは例えばこんなものです。

- **勤怠管理の自動化**: 部門25名の月次勤怠チェック。1人あたり約15分の手作業×25名 → コマンド1回・数分で完了。
- **グループウェアのCLI化**: スケジュール確認、稟議承認、施設予約。毎日のブラウザ操作10分くらい → コマンドで秒単位。
- **スキル管理システムの一括処理**: 半期ごとの全社員診断状況の取得・更新・提出を一発で。
- **スライド自動生成**: Markdown を書いたら自社テンプレートに沿った PowerPoint が出てくる仕組み。
- **MCPサーバーの構築**: Microsoft To Do と AI 会話を繋いで、会話の中でタスクを作成・完了できるように。

いずれも特別なことはしていません。Claude Code に「こういうことをしたい」と伝え、対話を重ねながら作って、動作確認して、運用に乗せました。Python の業務利用経験が多少あれば、誰でも同じことができます。

3ヶ月の間に、こうしたツールが17個になりました。

## 時間が空いた、はずだった

ツールが増えるにつれて、確かに時間は空きました。月次で半日かかっていた業務が10分で終わる、というのが積み重なれば、当然そうなります。

私は「空いた時間で、自分にしかできないことに集中するんだ」と考えていました。経営方針の検討、メンバーの育成、顧客との関係構築、外部のトレンドのキャッチアップ。管理職としてやるべきだけど、後回しにしがちな領域です。

ところが、しばらくして奇妙な感覚に気づきました。

**「自分は本当に、空いた時間を意味あることに使えているのだろうか？」**

体感としては「使えている気がする」。でも、それを言語化しようとすると曖昧でした。誰かに「先週、自動化で空いた時間を何に使いましたか？」と聞かれたら、説得力を持って答えられない。

3ヶ月で17個のツールを作ったというストーリーは話せます。でも、**「そのおかげで何が変わったか」を時間という尺度で語れない**。これは、自分にとって思った以上に居心地の悪い感覚でした。

## 「時間が見えない」という問題

考えてみると、私たちは**自分の時間の使い方を、ほぼ感覚でしか把握していません**。

「今週はミーティングが多かった気がする」
「最近 Slack 見すぎてる気がする」
「集中できる時間が朝だった気がする」

全部、気がするです。検証する手段がない。

業務時間の家計簿、というと大袈裟ですが、**そもそも自分が何にどれだけ時間を使っているかという「収支」が見えていない**状態で、「時間の使い方を改善しよう」と言っているわけです。家計簿なしで節約しようとしているのと同じです。

決算書なしで経営判断をする会社はありません。でも自分の時間に関しては、それをやってきたのだと気づきました。

## WorkTracker を作った

シンプルなものを作りました。

- Windows のタスクスケジューラで、1分ごとにアクティブウィンドウ（フォアグラウンドのプロセス名とタイトル）を CSV に記録する
- 別途、Linux/WSL 側で Python スクリプトを動かして、CSV を業務カテゴリ別に集計する
- カテゴリ分類は設定ファイル（JSON）で自由に定義できる

それだけです。クラウドにアップしないし、通知もしないし、ゲーミフィケーションもしません。**ただ淡々と「いつ、何を、見ていたか」を記録して、「合計でどうなっているか」を表示するだけ**。

仕組みは身も蓋もないほどシンプルですが、これを2週間運用してみて、思っていた以上のことが見えてきました。

## 見えてきたもの

実際の数字は出せませんが、傾向として4つほど共有します。

### 1. メールチェックに思った以上の時間を使っていた

「ちょこちょこ確認する」を1日に何回もやっていました。1回あたりは数分でも、合計すると平気で1時間を超えます。これを「短時間で済んでいるから問題ない」と思い込んでいました。

→ メールチェックの時間帯を朝・昼・夕方の3回に決めて、それ以外はOutlookを開かないようにしました。結果として、集中時間が増えました。

### 2. 集中できる時間帯が、思っていたのと違った

私は朝型だと思っていました。だから、難しい仕事は午前中に入れるようにしていました。

データで見ると、**実際に集中している（VS Codeとブラウザを行き来せず、同じファイルを長く触っている）時間帯は、午後の早い時間でした**。午前中は会議とメッセージのやり取りに細切れに使っていて、深い思考はあまりできていませんでした。

→ 重要な思考作業を午後の早い時間に寄せるようにしました。

### 3. 「短い会議」が積もっていた

15分や30分の会議は「軽い」と思いがちです。でも数を集めると、長時間ミーティングよりはるかに時間を食います。しかもそれぞれの前後に文脈切り替えのコストが乗ります。

→ いくつかの定期ミーティングを思い切って隔週に変更しました。必要なら個別にチャットすれば足ります。

### 4. アイドル時間が多い

「画面の前にいるけど何もしていない時間」が、想像より多くありました。サボっているわけではなく、**作業と作業の間の「次に何をやろうかな」という空白**です。

これは見えなかった頃は気にもならなかったのですが、データで見ると、ここを減らすだけで実稼働時間がだいぶ増えることが分かりました。

→ 「次のアクション」をその場でメモする習慣をつけて、意思決定の迷いを減らしました。

## 自動化と自己観察は、対になっている

ここで気づいたのは、**自動化と自己観察は、片方だけでは効かない**ということです。

自動化だけやっていると、「時間が空いた感」はあるけれど、それを何に使っているかは見えない。
自己観察だけやっていると、「ここを改善したい」とは思っても、改善する余裕の時間はない。

両方を回して初めて、こういうループが起きます。

```
自動化 → 時間が空く →
       時間の使い方を観察 →
       「ここに使うべき」という発見 →
       次の改善アクション →
       新たな自動化 or 行動の変化 →
       さらに時間が空く...
```

このループの根っこには、**「時間は唯一、増やせない資源だ」** という前提があります。お金は稼げる。スキルは学べる。知識は買える。でも1日24時間を25時間にすることは、誰にもできません。

だからこそ、その24時間をどう使うかは、本来もっと意識的な意思決定であるべきで、その意思決定にはデータが必要だ、というのが今の私の結論です。

## 公開しました

WorkTracker は、社内固有の部分を抜いた汎用版を MIT ライセンスで公開しています。

https://github.com/yuki-katakami/worktracker

- Windows 側: PowerShell + Task Scheduler（依存ゼロ）
- Linux/WSL/macOS 側: Python 3.8+ stdlib のみ（依存ゼロ）
- カテゴリ分類は JSON 設定ファイルで完全カスタマイズ
- ローカルファイルにのみ保存、ネットワーク通信なし

セットアップは README と SETUP.md を見れば 30 分くらいでできます。

特に、

- AI ツールで業務を自動化していて
- でも空いた時間を有効活用できているか不安な人
- 自分の時間の使い方を、感覚ではなくデータで知りたい人

に向いています。逆に、ゲーム要素や通知や「集中モード」のような機能は一切ありません。**ただ記録して、ただ集計するだけ**のシンプルなツールです。

## おわりに

私は AI 時代の働き方を「**自動化で空いた時間を、自分にしかできないことに使う**」と捉えています。きれいな言葉ですが、これを実行するのは想像以上に難しい。なぜなら、

- 「自分にしかできないこと」は、自分自身にも言語化できていない
- 「空いた時間」も、自分自身では正確に把握できていない

からです。両方とも、**観察しないと見えてきません**。

WorkTracker はその「観察」のための道具です。何かを劇的に変えてくれるわけではありませんが、**「自分の時間がどこに行っているか分かる」という事実だけで、行動が変わります**。家計簿をつけ始めた瞬間に支出が変わるのと同じです。

もし、AI ツールでせっせと自動化しているけど「で、それで何が変わったんだっけ？」が答えにくいと感じている人がいたら、試してみてください。
